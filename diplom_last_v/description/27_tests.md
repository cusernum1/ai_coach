# tests/test_core_metrics.py — Автоматические тесты

## За что отвечает файл

Тесты проверяют что математические функции из `core/metrics.py` работают правильно. Запускаются командой `pytest` — без базы данных и без LLM.

## Как запустить

```bash
cd diplom_last_v
source .venv/bin/activate
pytest tests/ -v
```

### Тест 1: границы wellness_score

```python
def test_wellness_score_bounds():
    assert wellness_score(1, 10) == 95
    assert wellness_score(10, 1) == 5
    # out-of-range clipping
    assert wellness_score(-5, 20) == 95
```
- `assert` — утверждение. Если условие `False` → тест падает с ошибкой.
- Проверяем граничные значения: минимальная усталость + максимальный сон = 95.
- `wellness_score(-5, 20)` — значения вне диапазона должны зажиматься (clamp) → результат как при (1, 10).

**Почему 95, а не 100?** При усталости=1: `(10-1)*5 = 45`. При сне=10: `10*5 = 50`. Итого 95.

### Тест 2: метки самочувствия

```python
def test_wellness_label_ranges():
    assert wellness_label(90).startswith("🟢")
    assert wellness_label(60).startswith("🟡")
    assert wellness_label(40).startswith("🟠")
```

`.startswith("🟢")` — проверяем что строка начинается с зелёного кружка, не проверяем точный текст. Это позволяет изменить текст метки не ломая тест.

### Тест 4: математика ACWR

```python
def test_acwr_math():
    # Равная нагрузка: 7 дней = 28 дней → ACWR = 1.0
    assert acwr([5, 5, 5, 5, 5, 5, 5], [5] * 28) == 1.0
    
    # Скачок: среднее за 7 дней = 10, среднее за 28 = 5 → 10/5 = 2.0
    assert acwr([10] * 7, [5] * 28) == 2.0
    
    # Нет данных → None
    assert acwr([], [5]) is None
```
`[5] * 28` — создаёт список из 28 пятёрок: `[5, 5, 5, ..., 5]`.

Проверяем три случая: норма, скачок нагрузки, отсутствие данных.

## Утилиты (app/bot/utils.py) — тесты chunk_text и money

Эти тесты проверяют вспомогательные функции бота.

### Тест 7: короткий текст возвращается без изменений

```python
def test_chunk_text_short():
    assert chunk_text("hello") == ["hello"]
```
`chunk_text("hello")` — если текст короче лимита (4096 символов), возвращается список из одного элемента.

Проверяем что функция **не ломает короткие тексты** — не режет их и не добавляет ничего лишнего.

---

### Тест 8: длинный текст делится на части

```python
def test_chunk_text_splits_long():
    block = "x" * 2000
    long_text = block + "\n\n" + block + "\n\n" + block
    parts = chunk_text(long_text, limit=3800)
    assert len(parts) >= 2
    for p in parts:
        assert len(p) <= 3800
```
- `"x" * 2000` — создаём строку из 2000 символов "x"
- `block + "\n\n" + block + "\n\n" + block` — три блока по 2000, разделённые двойным переносом строки. Итого ~6004 символа
- `chunk_text(long_text, limit=3800)` — передаём кастомный лимит 3800 (чуть меньше чем дефолтные 4096 Telegram)
- `len(parts) >= 2` — текст должен разделиться минимум на 2 части
- `for p in parts: assert len(p) <= 3800` — каждая часть не превышает лимит

Функция умно делит по двойным переносам строк (`\n\n`) — не рвёт абзацы посередине.

### Тест 9: жёсткое разрезание сверхдлинного блока

```python
def test_chunk_text_hard_split():
    big_block = "a" * 5000
    parts = chunk_text(big_block, limit=3800)
    assert len(parts) == 2
    assert len(parts[0]) == 3800
    assert len(parts[1]) == 1200
```
- `"a" * 5000` — строка из 5000 символов без единого переноса строки
- Такой блок нельзя разрезать по абзацам — нет разделителей
- Функция должна резать **жёстко**: первые 3800 символов → первая часть, оставшиеся 1200 → вторая
- `len(parts) == 2` — ровно 2 части
- `len(parts[0]) == 3800` — первая часть ровно на лимите
- `len(parts[1]) == 1200` — остаток: 5000 − 3800 = 1200

### Тест 10: форматирование рублей

```python
def test_money_rub():
    assert money(100000) == "1 000.00 ₽"
```
- `money(100000)` — принимает сумму в **копейках** (как Telegram Payments)
- 100 000 копеек = 1 000 рублей
- `"1 000.00 ₽"` — форматированная строка: пробел как разделитель тысяч, две цифры после запятой, знак рубля

Telegram Payments работает в наименьших единицах валюты (копейки/центы), поэтому суммы хранятся как целые числа — 100 рублей = 10000 копеек.

---

### Тест 11: форматирование долларов

```python
def test_money_usd():
    assert money(499, "USD") == "4.99 USD"
```
- `money(499, "USD")` — второй аргумент — код валюты
- 499 центов = 4.99 доллара
- Для USD нет разделителя тысяч (число небольшое), формат `"4.99 USD"`

Функция `money()` умеет форматировать разные валюты. Для ₽ ставит символ перед числом, для USD — код после числа.

## Зачем нужны тесты

1. **Уверенность**: если изменил формулу — тест покажет что что-то сломалось
2. **Документация**: тест показывает как должна работать функция
3. **Защита от регрессий**: старые баги не возвращаются

## Почему тестируем метрики и утилиты, но не handlers

В дипломном MVP тесты написаны для «чистых» функций без внешних зависимостей:
- Не требуют базы данных
- Не требуют Telegram-токена или LLM-ключа
- Запускаются мгновенно (миллисекунды)
- Легко проверить правильность вручную

Всего **11 тестов** (6 метрики + 5 утилиты):

| # | Тест | Что проверяет |
|---|------|--------------|
| 1 | `test_wellness_score_bounds` | Формула wellness_score, зажимание значений |
| 2 | `test_wellness_label_ranges` | Цветовые метки по диапазонам |
| 3 | `test_training_load_sums_rpe` | Сумма RPE-нагрузки |
| 4 | `test_acwr_math` | Формула ACWR (норма, скачок, нет данных) |
| 5 | `test_acwr_zone_labels` | Текстовые метки зон нагрузки |
| 6 | `test_training_monotony` | Монотонность (деление на 0, нормальный случай) |
| 7 | `test_chunk_text_short` | Короткий текст не разрезается |
| 8 | `test_chunk_text_splits_long` | Длинный текст режется по абзацам |
| 9 | `test_chunk_text_hard_split` | Жёсткое разрезание без разделителей |
| 10 | `test_money_rub` | Форматирование рублей из копеек |
| 11 | `test_money_usd` | Форматирование долларов из центов |

Для полноценного покрытия нужны интеграционные тесты (с тестовой БД) и моки Telegram API — это за рамками дипломного проекта.

## Ключевые термины

- **pytest** — популярный Python-фреймворк для тестирования
- **`assert`** — утверждение: если False → тест падает
- **Unit test** — тест одной функции в изоляции
- **Граничные условия (boundary testing)** — тест крайних значений (0, min, max)
- **Regression testing** — проверка что старые баги не вернулись
